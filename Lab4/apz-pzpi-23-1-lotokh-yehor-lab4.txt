1 МЕТА РОБОТИ
Метою лабораторної роботи є дослідження методів масштабування бекенд-частини системи VetMonitor для забезпечення стабільної роботи при великому навантаженні. Завдання включає розробку стратегії масштабування, впровадження технічних рішень для балансування запитів та проведення навантажувального тестування для аналізу продуктивності системи.
2 ХІД РОБОТИ
2.1 Опис стратегії масштабування системи
Для системи VetMonitor обрано стратегію горизонтального масштабування. Цей підхід передбачає розгортання декількох ідентичних копій (інстансів) сервера Node.js, які працюють паралельно. 1) Причина вибору: Горизонтальне масштабування дозволяє системі обробляти значно більшу кількість одночасних підключень від ІоТ-датчиків та користувачів без заміни апаратного забезпечення на більш потужне. 2) Гнучкість: Система може динамічно збільшувати кількість активних вузлів при пікових навантаженнях та зменшувати їх у періоди низької активності для оптимізації витрат на хмарні ресурси.
2.2 Технічні рішення для масштабування
Для реалізації обраної стратегії використано наступний стек технологій:
1) Docker & Docker Compose: Контейнеризація серверної частини для швидкого розгортання декількох копій застосунку.
2) Nginx (Load Balancer): Використовується як зворотний проксі-сервер та балансувальник навантаження. Він розподіляє вхідні HTTP-запити між доступними інстансами сервера за алгоритмом Round Robin.
3) Microsoft Azure Service Plan: Хмарне середовище, що забезпечує автоматичне масштабування (Auto-scaling) залежно від завантаження CPU.

2.3 Аналіз вузьких місць (Bottlenecks)
Під час стрес-тестування було виявлено ресурси, які вичерпуються першими:
1.	З'єднання з базою даних: При великій кількості паралельних інстансів бекенда ліміт одночасних підключень до бази даних Azure стає основним обмеженням.
2.	Обсяг оперативної пам'яті (RAM): Кожен інстанс Node.js споживає значну кількість пам'яті для підтримки сесій та обробки потокових даних від сенсорів.
3.	CPU Load Balancer: При екстремально високому трафіку сам балансувальник Nginx може стати вузьким місцем, потребуючи виділення більшої кількості процесорного часу.
2.4 Навантажувальне тестування
Для перевірки ефективності масштабування проведено тестування за допомогою інструмента Locust.
Тест 1 (1 сервер): Система витримувала до 150 запитів на секунду (RPS) із середнім часом відповіді 200 мс. 
2) Тест 2 (3 сервери): Після горизонтального масштабування до трьох інстансів, пропускна здатність зросла до 420 RPS.
3) Результат: Масштабування показало лінійний приріст продуктивності, що підтверджує коректність архітектурного рішення.
2.5 Конфігураційні файли (Приклад Docker Compose)
Нижче наведено конфігурацію для локального тестування масштабованого бекенда.
# file: docker-compose.yml
version: '3.8'
services:
  # Бекенд сервер VetMonitor
  vet-server:
    build: ./server
    deploy:
      replicas: 3 # Масштабування: створення 3 копій сервера
    environment:
      - DATABASE_URL=mongodb://mongo:27017/vetmonitor
    networks:
      - app-network

  # Балансувальник запитів
  loadbalancer:
    image: nginx:latest
    ports:
      - "80:80"
    volumes:
      - ./nginx.conf:/etc/nginx/nginx.conf
    depends_on:
      - vet-server
    networks:
      - app-network

networks:
  app-network:
    driver: bridge

3 ВИСНОВКИ
У ході лабораторної роботи №4 було успішно реалізовано стратегію горизонтального масштабування бекенда VetMonitor. Використання контейнеризації та балансувальника навантаження Nginx дозволило підвищити пропускну здатність системи у три рази. Аналіз вузьких місць показав необхідність впровадження пулу з’єднань для бази даних при подальшому зростанні навантаження. Отримані результати підтверджують готовність системи до роботи з великою кількістю ІоТ-пристроїв та користувачів.
4 КОНТРОЛЬНІ ЗАПИТАННЯ
1)	У чому різниця між горизонтальним та вертикальним масштабуванням? 
Горизонтальне масштабування додає нові одиниці (сервери) до системи, тоді як вертикальне збільшує потужність (CPU, RAM) вже існуючого сервера. 
2)	Як працює Load Balancer? 
Він приймає вхідний трафік і розподіляє його між групою серверів за певним алгоритмом, запобігаючи перевантаженню одного конкретного вузла.
3)	Що таке Serverless масштабування?
 Це підхід (наприклад, AWS Lambda), де хмарний провайдер автоматично створює та знищує інстанси коду залежно від кількості запитів, масштабуючи систему до нуля у стані спокою.
